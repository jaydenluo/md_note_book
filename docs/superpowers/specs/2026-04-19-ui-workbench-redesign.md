# Muse Note UI Workbench Redesign

**Date:** 2026-04-19

**Status:** Approved for planning

**Source:** Selected from Superpowers brainstorm option `A. Calm Workbench`

## Goal

在不破坏现有文档编辑与提醒卡片功能的前提下，重整 Muse Note 的主界面，使其从“能用的三栏工具”升级为“稳定、克制、适合长期使用的桌面工作台”。

本次改造的核心目标：

- 提升信息层级清晰度
- 让三栏布局更像统一工作区，而不是三块独立面板
- 强化“导航 / 定位 / 编辑”三段式认知
- 保持高效，不引入营销化或装饰性过强的设计
- 为后续 spec -> plan -> implementation 留出稳定基础

## Product Direction

选定方向为 `Calm Workbench`。

这意味着：

- 保留桌面端的三栏结构
- 保留提醒模式的独立右侧工作区
- 整体视觉采用克制、偏专业、低噪音的工作台风格
- 以少量青绿色作为焦点色，而不是使用多枚强调色
- 优先改善框架层、导航层、列表层与编辑容器层

不采用：

- 高饱和度炫技风格
- 营销页式大横幅或 hero
- 过度卡片化的后台感
- 过密、过硬的开发者面板风格
- 过于极简以至于牺牲整理效率的沉浸写作风格

## Visual Thesis

界面应该看起来像“安静但有秩序的知识工作台”。

关键词：

- calm
- structured
- editorial
- desktop-native
- low-noise
- premium utility

视觉判断标准：

- 第一眼先看到结构，再看到颜色
- 品牌存在感来自版式和节奏，不靠大面积 Logo 装饰
- 长时间使用不累
- 空状态、分组状态、选中状态都能快速识别

## Color and Surface

建议色彩策略：

- 主体背景：浅冷灰到浅青灰渐变
- 面板底色：半透明白或深色半透明面板
- 焦点色：teal / cyan 系
- 危险操作：只在删除等高风险行为中使用红色

基础原则：

- 常态界面不依赖强烈色块分隔
- 选中态和焦点态使用同一套强调色语义
- 面板依靠轻边框、模糊和阴影建立层级
- 不同区域之间的差别来自“密度与结构”，而不是“颜色互打”

## Typography

文字应保持桌面工具感，而非海报感。

规则：

- 标题使用更紧的字距和更清晰的粗细层级
- 列表文字优先保证扫描速度
- 辅助信息统一变轻，不抢主标题
- 同屏内层级不超过 4 级

推荐层级：

- App / Panel 标题：18-24px，semibold 或 bold
- 区域说明：12-13px，uppercase 或低饱和正文
- 列表标题：14-15px，medium 或 semibold
- 辅助元信息：12-13px

## Information Architecture

### Desktop

桌面端维持三段式结构：

1. 左栏：全局范围与模式切换
2. 中栏：当前分类下的目录和文档定位
3. 右栏：当前主要工作面

其中：

- 左栏回答“我在哪个范围里工作”
- 中栏回答“我要打开哪个内容”
- 右栏回答“我正在处理什么”

### Mobile

移动端不追求完整复刻桌面三栏。

规则：

- 左栏改为抽屉
- 文档模式下优先显示当前文档或当前列表
- 提醒模式下保持单主区
- 保留一致的视觉语言，但减少面板嵌套感

## Panel Specs

### Left Sidebar

职责：

- 品牌识别
- 全局搜索入口
- 文档 / 提醒模式切换
- 分类切换
- 全局统计摘要
- 底部功能操作

设计要求：

- 顶部必须包含更明确的品牌区
- 搜索框要比旧版更稳定、更像工作区工具
- 模式切换控件应作为一级信息出现
- 分类列表从“普通行项”升级为“导航项”
- 选中分类必须一眼可见

行为要求：

- 分类 hover 时才暴露危险操作
- 新建分类按钮应靠近“分类”标题
- 分类数量统计在文档模式与提醒模式下语义不同，但视觉规则一致

### Middle Note List

职责：

- 展示当前范围下的目录与文档
- 承接新建文档、新建目录、拖拽整理
- 成为用户进入编辑态前的“定位区”

设计要求：

- 顶部有更完整的区域标题、数量摘要与操作按钮
- 目录应形成明显分组块，而不是一条条平铺
- 根级文档应有独立分区
- 空白区不能只是广告位占位，必须承担明确提示作用

交互要求：

- 目录折叠态与展开态过渡要平滑
- 当前选中文档高亮要清楚，但不能过于刺眼
- 拖拽进入根目录或目录时应有明显接受态
- 右键空白区新建的入口应保留

### Right Content Area

职责：

- 文档模式下承载标签栏与编辑器
- 提醒模式下承载卡片墙或提醒工作区

设计要求：

- 整个右栏应作为一块完整工作容器存在
- 标签栏与正文容器应视觉属于同一系统
- 空状态必须成熟、稳定，不再是临时占位感
- 编辑器外层应有明确边界，但不能抢内容本身的注意力

## Tab System

标签栏是桌面感关键部分。

要求：

- 默认状态轻量、清楚
- 活动标签应拥有强识别度
- 关闭按钮需要稳定的 hover 反馈
- 溢出标签应通过折叠菜单处理
- “关闭全部标签”要保留，但视觉上不喧宾夺主

标签栏目标不是炫，而是：

- 快速定位
- 快速切换
- 快速关闭

## Reminder Mode

提醒模式沿用现有独立工作区思路，但视觉上需要与文档模式属于同一套系统。

要求：

- 左栏继续保留
- 中栏隐藏
- 右栏成为完整的提醒工作台
- 卡片墙、筛选、搜索和新建按钮仍保留

提醒模式不应做成完全不同的产品，而应像同一应用中的另一种工作上下文。

## Motion

允许少量动效，但必须服务于层级和状态。

建议保留：

- 面板进入时的轻微过渡
- 标签切换和 hover 反馈
- 目录展开收起的平滑过渡
- 模态或浮层的稳定显隐

避免：

- 大幅位移动画
- 花哨缩放
- 高频闪动
- 与工作内容无关的装饰性运动

## Content Rules

文案必须偏工具化，而不是宣传化。

好的文案类型：

- 工作区
- 当前视图
- 文档
- 提醒
- 分类
- 新建文档
- 新建目录
- 这里还没有内容

避免：

- 口号式标题
- 情绪化说明文案
- 与操作无关的氛围句

## Constraints

本次 spec 明确限制：

- 不重构底层存储模型
- 不改写编辑器核心能力
- 不在本轮引入全新导航体系
- 不引入复杂动画系统
- 不做视觉语言完全推翻式重做

本次重整更偏：

- UI architecture refresh
- visual consolidation
- hierarchy cleanup
- workflow polish

## Non-Goals

以下内容不在本轮范围内：

- 全量设计系统建设
- 提醒功能规则改版
- 编辑器工具栏功能重做
- 富文本交互范式重构
- 文件树模型升级
- 多窗口或多工作区机制

## Validation Criteria

完成后应满足：

- 打开应用后，用户能立即理解三栏各自职责
- 左栏、中栏、右栏在视觉上属于同一系统
- 当前选中分类与当前文档更容易扫描
- 空状态不再显得临时和粗糙
- 标签栏具有更强桌面工作台感
- 提醒模式与文档模式风格统一，但结构差异明确
- 桌面与移动端都能正常工作
- 不影响现有创建、切换、拖拽、右键和标签关闭等核心行为

## Next Step

下一步应基于本 spec 生成 implementation plan，按以下顺序拆解：

1. 全局骨架与视觉容器
2. 左栏导航与模式区
3. 中栏目录 / 文档列表重整
4. 标签栏与右侧空状态
5. 提醒模式视觉统一
6. 验证与回归测试
