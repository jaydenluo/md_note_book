# 数据库锁定问题修复说明

## 问题描述

在应用运行过程中，经常出现以下错误：
```
storage.ts:554 保存分类数据失败: error returned from database: (code: 5) database is locked
```

## 问题原因分析

1. **并发数据库访问**：多个操作同时尝试访问数据库，导致 SQLite 锁定
2. **数据库初始化竞争**：多个 `initDb()` 调用同时执行，可能创建多个数据库连接
3. **事务管理不当**：写操作没有正确串行化，导致死锁

## 修复方案

### 1. 添加数据库初始化互斥锁

在 `src/services/storage.ts` 中添加了 `withDbInit` 函数：

```typescript
// 数据库初始化互斥锁，避免并发初始化
let initQueue: Promise<void> = Promise.resolve();

function withDbInit<T>(fn: () => Promise<T>): Promise<T> {
  const next = initQueue.then(() => fn());
  // 确保无论成功失败都释放队列
  initQueue = next.then(() => undefined).catch(() => undefined);
  return next;
}
```

### 2. 修改 initDb 函数

将 `initDb` 函数包装在 `withDbInit` 中，确保数据库初始化操作是串行的：

```typescript
async function initDb(customPath?: string): Promise<Database | null> {
  if (!isTauri) return null;
  
  return withDbInit(async () => {
    // ... 原有的初始化逻辑
  });
}
```

### 3. 改进事务回滚处理

修复了空的 catch 块，添加了适当的错误处理：

```typescript
try { 
  await database.execute('ROLLBACK'); 
} catch (rollbackError) {
  console.warn('回滚事务失败:', rollbackError);
}
```

## 修复效果

1. **避免并发初始化**：数据库连接初始化现在是串行的，不会出现多个连接竞争
2. **保持写操作串行化**：所有数据库写操作仍然通过 `withDbWrite` 串行执行
3. **独立锁机制**：初始化锁和写锁是独立的，不会相互阻塞
4. **更好的错误处理**：事务回滚失败时会有适当的日志记录

## 测试验证

创建了 `src/test/databaseLockTest.ts` 测试文件，验证：
- `withDbWrite` 正确串行化写操作
- `withDbInit` 正确串行化初始化操作
- 两种锁机制独立工作，不会相互干扰

## 注意事项

1. **性能影响**：初始化操作现在是串行的，可能略微增加启动时间
2. **错误恢复**：如果数据库初始化失败，应用会优雅降级到 LocalStorage 模式
3. **监控建议**：建议监控数据库锁定错误的频率，确保修复效果
